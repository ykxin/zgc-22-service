## 五、AI 责任判定引擎 `analyze_dispute()`

### 5.1 服务质量纠纷（`service_quality`）

| # | 验证项 | 状态 | 说明 |
|---|--------|------|------|
| 5.1.1 | 验收得分 < 60 → 从业者责任 | ✅ | 已实现，建议补服务或退50%费用 |
| 5.1.2 | 打卡完成率 < 70% → 从业者责任 | ✅ | 已实现，建议按比例退费 |
| 5.1.3 | 验收得分 ≥ 80 → 雇主责任（恶意投诉） | ✅ | 已实现，建议驳回纠纷 |
| 5.1.4 | 中间分段（60-79）→ 双方责任 | ✅ | 已实现，建议协调补做 |
| 5.1.5 | 验收得分为 `None` 时的处理 | ⚠️ | `acceptance_score < 60` 在 None 时会跳过，但 `completion_rate` 分支可兜底；逻辑不够清晰 |

### 5.2 薪资纠纷（`salary`）

| # | 验证项 | 状态 | 说明 |
|---|--------|------|------|
| 5.2.1 | 验收得分 ≥ 70 → 雇主应足额支付 | ✅ | 已实现 |
| 5.2.2 | 验收不达标 → 按比例支付 | ✅ | 已实现，计算公式 `price * score / 100` |
| 5.2.3 | `acceptance_score` 为 `None` 时按比例计算结果为 0 | ⚠️ | `None * 0 / 100` 会触发 TypeError，或因 `and` 短路返回 both 分支，但 both 分支中 `(acceptance_score or 0)` 使得应付金额为 0，对从业者不公平 |
| 5.2.4 | 从业者主动发起薪资纠纷的场景 | ⚠️ | 判定逻辑未区分发起方，从业者投诉克扣薪资与雇主投诉服务费争议走同一分支 |
